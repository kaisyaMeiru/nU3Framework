
### A. 컨트롤 Wrapping 
  DevExpress 컨트롤을 Wrapping(감싸기/상속)하는 것은 대규모 SI 프로젝트나 엔터프라이즈 프레임워크(nU)에서 "선택이 아니라 필수"에 가깝습니다.
  단순히 "DevExpress 기능을 그대로 쓴다"면 굳이 래핑할 필요가 없겠지만, 장기적인 유지보수 효율성을 고려한다면 래핑에는 명확하고 강력한 이유가 있습니다.
  그 5가지 핵심 이유(Why)와 실전 전략(How)을 설명해 드립니다.
  ---
  1. 표준화 강제 (Standardization)
  DevExpress 컨트롤은 기능이 너무 많고 자유도가 높습니다. 개발자 10명에게 그리드를 만들라고 하면 10가지 다른 모양의 그리드가 나옵니다.
   * 래핑의 효과:
       * nUGridControl을 만들고 생성자에서 UseEmbeddedNavigator = false, ShowGroupPanel = false 등을 기본값으로 박아버리면, 모든 화면의 그리드가 통일된 모양을 갖게 됩니다.
       * "우리 프로젝트는 모든 날짜 입력(DateEdit)을 yyyy-MM-dd 형태로만 쓴다"는 규칙을 컨트롤 내부에서 강제할 수 있습니다.

  2. 프레임워크 기능 주입 (Framework Integration)
  DevExpress 컨트롤은 nU 프레임워크의 존재를 모릅니다. 하지만 래핑 컨트롤은 알 수 있습니다.
   * 래핑의 효과:
       * 자동 번역: nUButton의 Text 속성이 바뀔 때마다 프레임워크의 다국어 사전(Dictionary)을 조회해서 자동으로 번역된 텍스트를 출력하게 만들 수 있습니다.
       * 권한 제어: nUButton에 AuthID="btnSave"라고 주면, 컨트롤이 로드될 때 자신의 권한을 체크해서 Enabled=false로 스스로 잠그게 할 수 있습니다.

  3. 교체 용이성 (Vendor Independence)
  극단적인 예로, 5년 뒤에 DevExpress 라이선스 비용이 10배 올라서 다른 벤더(Infragistics 등)로 바꿔야 한다면?
   * 래핑의 효과:
       * 소스 코드에 DevExpress.XtraGrid.GridControl이라고 직접 써놨다면 프로젝트 전체 소스 수천 개를 다 고쳐야 합니다.
       * nU.Controls.GridControl로 래핑해서 썼다면, 래핑 클래스 내부의 상속 대상만 바꾸고 내부 연결 로직만 수정하면(물론 쉽지는 않지만) 업무 화면 코드는 최소한의 수정으로 보호받을 수 있습니다.

  4. 확장 기능 추가 (Extension) 
   * 래핑의 효과:
       * `ToExcel()` 메서드 추가: 모든 그리드에 "엑셀 다운로드"라는 메서드를 추가하고, 공통적인 디자인(헤더, 색상)을 적용해서 내보내게 할 수 있습니다.
       * 개인화 저장/복원: 앞서 논의한 WorkspaceManager나 설정 저장 로직을 모든 컨트롤의 OnLoad, OnDispose에 심어둘 수 있습니다.


  5. 복잡성 은닉 (Simplicity)
  DevExpress는 설정이 복잡합니다. 초급 개발자에게 "그리드에 콤보박스 컬럼 하나 추가해"라고 하면 RepositoryItem을 만들고 연결하는 과정을 어려워합니다.
  
### B. 사용자 개인화 (Personalization) 저장
- [ ] **레이아웃 저장**: 사용자가 변경한 그리드 컬럼 순서, 너비, 정렬, 그룹핑 상태가 세션 종료 후에도 유지되는가? (XML/DB 저장)
- [ ] **즐겨찾기/기본값**: 검색 조건이나 자주 쓰는 값을 기본값으로 저장하는 기능이 있는가?

### C. UI 생산성 도구 (Productivity Tools)
- [ ] **일괄 작업**: 엑셀 업로드/다운로드 기능이 모든 그리드에 표준으로 제공되는가? 
- [ ] **대량 데이터 처리**: nU의 그리드가 Server Mode(Virtual Mode)를 지원하여 대량 데이터 조회 시 성능 문제가 없는지? (DataWindow는 RetrieveAsNeeded 지원)

### D. 프레임워크 표준 (Framework Standards)
- [ ] **메시지 처리**: `IMsgContainer`와 같은 표준 메시지 처리기가 `BaseWorkControl`이나 `nU` 서비스에 구현되어 있는가?
- [ ] **권한 제어**: 버튼/컨트롤 단위의 권한 제어(`CanCreate`, `CanPrint` 등)가 UI 요소의 `Enabled`/`Visible` 속성과 자동 연동되는가?

## 5. 결론 및 제언 (Recommendation)

**nU**는 **DevExpress** 기반의 세련된 UI와 현대적인 아키텍처를 가지지만, DataWindow가 제공하던 **"동적 화면 생성"** 및 **"자동화된 SQL 바인딩"** 기능은 기본적으로 부재합니다.

**성공적인 마이그레이션을 위한 제언:**
1.  **Generic Search Popup 개발**: `XFindGrid`를 대체할 수 있는, 메타데이터 기반의 공통 검색 팝업을 우선 개발하십시오.
2.  **Layout Persistence 구현**: `BaseWorkControl` 레벨에서 폼 종료 시 그리드 레이아웃을 XML로 자동 저장하고, 로드 시 복원하는 기능을 표준화하십시오.
3.  **SQL 추출 및 API화**: DataWindow 마이그레이션의 핵심은 UI가 아니라 **내장된 SQL의 비즈니스 로직 분리**입니다. 이를 자동화하거나 체계적으로 수행할 전략이 필수적입니다.
